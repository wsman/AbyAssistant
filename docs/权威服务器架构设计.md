# 权威服务器架构设计

> 目标：将当前 `src/` 目录下的 CLI / agent / swarm / remote 编排体系，重构为“**可连接远端权威服务器的客户端**”；由远端服务端统一承载控制平面，汇总多个客户端、多个用户、多个 agent 的协作状态、任务状态、审批状态、审计状态和共享交互状态。

---

## 1. 设计目标

本方案的核心目标不是把所有执行都搬到服务器，而是建立一个**逻辑上全局唯一的服务端控制平面**，使当前 `src/` 作为客户端接入，满足以下要求：

1. **多客户端共享同一组权威真相**
   - 同一用户的多个客户端
   - 多个用户参与同一团队/项目/会话
   - 多种 agent 编排形式并存

2. **服务端统一承载控制面闭环**
   - 统一请求分类路由
   - 统一前置行为裁决
   - 统一状态收束
   - 统一受控回写
   - 统一受控提交
   - 统一事件记录与恢复

3. **客户端保留运行时能力**
   - 本地 Bash / REPL / worktree / tmux / iTerm2 / in-process teammate
   - 远端 CCR agent
   - 本地工具调用与 UI 交互

4. **运行时能力不能直接写入权威状态**
   - 所有执行结果、摘要、诊断、证据都必须先经过受控回写通道

5. **客户端支持离线模式与在线模式双模运行**
   - **离线模式**：保持 `src/` 当前已有的本地优先功能与运行语义
   - **在线模式**：连接权威服务器，在服务端工作区体系中运行
   - 每个用户在在线模式下至少拥有一个**个人工作区**，并可加入一个或多个**共享工作区**

---

## 2. 非目标

本架构**不追求**以下目标：

1. 不把所有工具执行都集中到服务端。
2. 不让服务端直接替代 tmux pane、REPL VM、本地文件系统和本地 secrets。
3. 不把所有原始 transcript、完整源码、完整 shell 输出默认上送到服务端。
4. 不要求第一阶段就实现完整集群化；但需要从一开始保留事件溯源与恢复接口。
5. 不强迫所有用户始终在线；离线模式必须继续兼容当前 `src/` 的主要本地能力。

---

## 3. 当前系统的控制面现状

从现有 `src/` 看，控制面能力已经存在，但分散在多个局部状态里：

- `AppState.tasks`：本地任务状态
- `AppState.teamContext`：本地团队共享视图
- `team file`：团队 durable roster
- `teammateMailbox.ts`：文件式协作消息
- `task-notification`：任务结果回流机制
- `LocalAgentTask / RemoteAgentTask / InProcessTeammateTask`：不同任务后端
- `AsyncLocalStorage`：同进程 agent 身份隔离
- `src/bridge/remoteBridgeCore.ts` / `src/server/directConnectManager.ts`：已有远端桥接和连接管理骨架

因此当前系统并不是“没有控制面”，而是：

> **控制面存在，但还没有一个逻辑上全局唯一的服务端权威状态结构。**

本设计的核心，就是把这些分散控制信息**上收为服务端权威状态**，而把客户端本地状态降级为投影与运行时缓存。

---

## 4. 离线模式与在线模式的明确区分

这一节是整个设计的主轴，不再把 `offline` / `online` 视为附属选项，而是视为两种明确的运行制度。

### 4.1 离线模式（offline）

离线模式的定义不是“完全不联网”，而是：

> **不依赖权威服务器，也不以服务端状态作为控制面真相。**

离线模式下：

- 不要求 `session.register`
- 不要求 `workspace.bind`
- `AppState.tasks / teamContext / inbox` 仍然可以作为本地真相
- `TeamFile` 仍可作为团队持久化来源
- `teammateMailbox.ts` 仍可作为协作消息主通道
- 审批、任务通知、消息投递继续走本地闭环
- 本地 Bash / REPL / tmux / iTerm2 / worktree / in-process teammate 继续按当前语义运行

离线模式适用场景：

- 单机使用
- 本地实验
- 不需要跨客户端共享的个人任务
- 暂时无法连接权威服务器的环境

### 4.2 在线模式（online）

在线模式的定义是：

> **客户端连接权威服务器，并以服务端状态作为共享控制面的唯一真相来源。**

在线模式下：

- 必须先完成 `session.register`
- 必须绑定一个 `activeWorkspaceId`
- 所有共享控制面对象都必须归属某个 `workspaceId`
- 本地 `AppState` 不再作为共享真相，只是 projection cache + runtime state
- 所有跨客户端共享对象都通过服务端创建、更新、投递和审计

在线模式适用场景：

- 多用户协作
- 多客户端共享任务面板
- 服务端统一审批与治理
- 服务端审计、恢复、事件回放
- 个人工作区到共享工作区的受控发布

### 4.3 双模共享与双模分离

两种模式共享的是**执行器层**，分离的是**控制面真相来源**。

共享：

- Bash
- REPL
- 本地文件编辑
- 本地 worktree
- tmux / iTerm teammate
- in-process teammate
- remote CCR agent 适配

分离：

- 离线模式：本地状态是真相
- 在线模式：服务端权威状态是真相

### 4.4 工作区视角

- 离线模式可以被视为运行在一个**本地离线工作区**中，但该工作区不在服务端注册。
- 在线模式必须运行在服务端工作区体系中。
- 在线模式下每个用户至少拥有一个**个人工作区**。
- 在线模式下多个用户可以加入同一个**共享工作区**。

## 5. 哪些功能应当加入在线模式

这一节只回答一个问题：

> 与当前本地 `src/` 相比，在线模式必须新增哪些功能，才能构成真正的权威服务器模式。

### 5.1 在线模式必须新增的顶层功能

在线模式至少必须新增以下功能：

1. **服务端会话注册与在线态**
   - `session.register`
   - `session.heartbeat`
   - presence / reconnect / disconnect 状态

2. **工作区体系**
   - 个人工作区
   - 共享工作区
   - 工作区目录
   - 当前活动工作区绑定 `activeWorkspace`
   - 工作区切换

3. **服务端权威任务注册表**
   - 任务创建
   - 任务状态机
   - 任务投影
   - 任务通知
   - 任务恢复

4. **服务端邮箱与协作消息**
   - 直接消息
   - 广播消息
   - 审批请求
   - 审批响应
   - task-notification

5. **服务端治理与审批**
   - permission approval
   - sandbox approval
   - plan approval
   - shutdown approval
   - mode change governance

6. **受控回写通道**
   - progress summary writeback
   - result summary writeback
   - transcript summary writeback
   - artifact / evidence ref writeback

7. **投影订阅与重同步**
   - team/task/mailbox/governance projection
   - WebSocket push
   - projection version
   - resync / replay / reconnect

8. **服务端事件记录与恢复**
   - append-only event log
   - snapshot
   - projection rebuild
   - crash recovery

9. **个人工作区向共享工作区发布**
   - `workspace.publish_from_personal`
   - 发布前审批
   - 目标工作区 membership 校验

10. **在线模式下的权限与隔离**
   - workspace membership
   - workspace role
   - team membership
   - server-side visibility boundary

### 5.2 在线模式功能矩阵

下表明确区分：哪些功能离线就有，哪些是在线必须新增，哪些是双模复用但真相来源不同。

| 功能 | 离线模式 | 在线模式 | 说明 |
| --- | --- | --- | --- |
| 本地 Bash / REPL / Edit / Write | 保留 | 保留 | 运行时能力，双模共享 |
| tmux / iTerm teammate | 保留 | 保留 | 执行器仍在本地，在线仅服务端化控制面 |
| in-process teammate | 保留 | 保留 | 同上 |
| 本地 worktree | 保留 | 保留 | 运行时能力，不等于控制面 |
| `AppState.tasks` 作为真相 | 是 | 否 | 在线下只能是 projection cache |
| `AppState.teamContext` 作为真相 | 是 | 否 | 在线下只能是 team projection |
| `AppState.inbox` 作为真相 | 是 | 否 | 在线下只能是 mailbox cache |
| `TeamFile` 作为团队真相 | 是 | 否 | 在线模式应降级为兼容层或缓存 |
| `teammateMailbox.ts` 作为主消息通道 | 是 | 否 | 在线模式应改为服务端 mailbox |
| `task-notification` 本地生成 | 是 | 否 | 在线模式应由服务端生成投影事件 |
| `session.register / heartbeat` | 否 | 必须 | 在线模式新增 |
| `workspace.list / bind / switch` | 否 | 必须 | 在线模式新增 |
| 个人工作区 | 否 | 必须 | 在线模式新增 |
| 共享工作区 | 否 | 必须 | 在线模式新增 |
| 跨客户端 presence | 否 | 必须 | 在线模式新增 |
| 服务端 task registry | 否 | 必须 | 在线模式新增 |
| 服务端 mailbox | 否 | 必须 | 在线模式新增 |
| 服务端 governance | 否 | 必须 | 在线模式新增 |
| projection / subscription | 否 | 必须 | 在线模式新增 |
| event log / snapshot / recovery | 本地弱化 | 必须 | 在线模式新增完整控制面能力 |
| 个人工作区向共享工作区发布 | 否 | 必须 | 在线模式新增 |

### 5.3 在线模式下必须服务端化的现有本地功能

下面这些在当前仓库里是本地优先结构，但在在线模式下必须服务端化：

1. `AppState.tasks`
   - 在线模式下不能继续作为任务真相
   - 只能作为服务端任务投影缓存

2. `AppState.teamContext`
   - 在线模式下不能继续作为团队真相
   - 只能作为 team projection

3. `AppState.inbox`
   - 在线模式下不能继续作为原始消息来源
   - 只能缓存已投递但未消费的服务端消息

4. `TeamFile`
   - 在线模式下不能继续承担 durable roster 真相
   - 应由服务端 TeamAggregate 替代

5. `teammateMailbox.ts`
   - 在线模式下不能继续承担主消息通道
   - 应由服务端 Mailbox Service 替代

6. `task-notification`
   - 在线模式下不应由本地任务后端各自定义通知行为
   - 应先写入服务端任务真相，再由服务端统一推送通知

7. 本地审批桥
   - 在线模式下不应继续把 plan approval / shutdown approval / sandbox approval 视为本地局部行为
   - 必须写入服务端 GovernanceAggregate

### 5.4 在线模式下不必新增、只需复用的功能

这些能力在在线模式中不需要重写执行器，只需重写控制面接入：

- Bash 执行器
- REPL VM
- 本地文件工具
- 本地 worktree
- tmux / iTerm pane 执行
- in-process teammate 执行
- Remote CCR 作为远端执行后端

它们应继续“执行”，但不能继续“直接写最终状态”。

## 6. 总体架构

### 6.1 客户端双模运行模型

客户端在运行期必须先确定当前模式：

1. **离线模式（offline）**
   - 不连接权威服务器
   - 当前 `src/` 的本地控制面继续生效
   - `AppState.tasks / teamContext / inbox` 等本地结构仍可作为本地真相
   - 适用于单机使用、离线执行、本地实验与不需要多人共享的场景

2. **在线模式（online）**
   - 客户端连接权威服务器
   - 所有共享控制面对象都归属某个 `workspaceId`
   - 本地 `AppState` 降级为服务端 projection cache + 本地运行时状态
   - 适用于多人协作、跨客户端同步、服务端审批与审计闭环场景

可将双模模型概括为：

```text
+-----------------------------------+
|          Client (src)             |
|-----------------------------------|
| Offline Mode                      |
| - 本地 AppState / tasks           |
| - 本地 teamContext / mailbox      |
| - 本地运行时能力                  |
|                                   |
| Online Mode                       |
| - session.register / heartbeat    |
| - activeWorkspace 绑定            |
| - 服务端 projection cache         |
| - 本地运行时能力 + 受控回写       |
+-----------------+-----------------+
                  |
            if online only
                  |
+-----------------v-----------------+
|        Authority Server           |
| Workspace / Team / Task / Gov     |
| Mailbox / Projection / Events     |
+-----------------------------------+
```

### 6.2 工作区模型

为了让在线模式下的多人协作、多客户端同步和个人/共享边界清晰成立，服务端必须引入**工作区（Workspace）**作为顶层命名空间。

工作区分为三类概念：

1. **本地离线工作区（Local Offline Workspace）**
   - 仅在离线模式下存在的概念性工作区
   - 不在服务端注册
   - 用于统一解释离线模式下的本地 task / team / inbox / runtime

2. **个人工作区（Personal Workspace）**
   - 在线模式下每个用户默认拥有的私有工作区
   - 用于承载个人任务、个人 agent、个人摘要、个人草稿与私有编排
   - 默认仅本人可见

3. **共享工作区（Shared Workspace）**
   - 在线模式下用于多人协作的工作区
   - 多个用户可通过 membership 关系加入同一工作区
   - 共享 team、task、mailbox、governance 与 projection

在线模式下必须满足以下规则：

- 所有共享控制面对象都必须挂载 `workspaceId`
- 工作区是 team / task / mailbox / governance / projection 的上层归属
- 用户登录在线模式后，默认进入自己的个人工作区，但可以切换到某个共享工作区

### 6.3 逻辑分层

```text
+-------------------------------+
|        Client (src)           |
| CLI / UI / Tools / Swarm      |
| Local Runtime / Remote Adapt  |
+---------------+---------------+
                |
         HTTP / WebSocket
                |
+---------------v---------------+
|      Authority Server         |
| 1. 接入与鉴权层               |
| 2. 请求分类路由层             |
| 3. 前置行为裁决层             |
| 4. 统一变更流水线             |
| 5. 受控回写与吸收映射层       |
| 6. 任务编排与治理层           |
| 7. 消息/审批/邮箱层           |
| 8. Projection / Subscription  |
| 9. Event Log / Snapshot       |
+---------------+---------------+
                |
     Controlled Runtime Boundaries
                |
+---------------v---------------+
| Runtime Capability Plane      |
| Local tools / tmux / REPL     |
| In-process agents / CCR       |
| External capability services  |
+-------------------------------+
```

### 6.4 权威服务器与运行时能力平面的关系

- 权威服务器负责：决定系统共享真相是什么。
- 运行时能力平面负责：执行具体工作并返回受控结果。
- 客户端负责：承接本地执行、展示投影、向服务端提交治理请求与回写请求。

换句话说：

> **客户端继续“干活”，服务端统一“裁决、记账、广播、恢复”。**

## 7. 服务端核心模块设计

### 7.1 接入与鉴权层（Gateway）

职责：

- 客户端注册与鉴权
- WebSocket 长连接管理
- HTTP 命令入口
- 心跳与 presence 更新
- 协议版本协商

建议接口：

- `POST /v1/client/register`
- `POST /v1/session/heartbeat`
- `WS /v1/stream`

关键输出：

- `sessionId`
- `clientId`
- `tenantId / workspaceId`
- 当前订阅游标 / projection version

### 7.2 请求分类路由层（Classification Router）

职责：

- 将客户端请求分为核心治理请求与状态回写请求
- 为不同请求打上 `commandId / requestId / idempotencyKey`
- 决定是否需要进入裁决层、变更流水线或回写通道

输入示例：

- `team.create`
- `agent.spawn`
- `task.stop`
- `governance.respond`
- `task.progress.writeback`
- `task.result.writeback`

### 7.3 前置行为裁决层（Adjudication Layer）

职责：

- 对治理请求输出：执行 / 改写 / 拒绝
- 执行能力清单校验
- 审批前置判定
- 风险/保护等级控制

### 7.4 统一变更流水线（Unified Change Pipeline）

职责：

- 对治理请求做规范化
- 校验领域约束
- 生成状态更新操作
- 将状态变更写入权威状态结构

### 7.5 受控回写与吸收映射层（Writeback Ingestion & Mapping）

职责：

- 接收来自客户端或运行时能力的受控结果
- 执行语义解释
- 将结果映射为合法状态更新操作或审计事件

输入示例：

- `task.progress.writeback`
- `tool.result.writeback`
- `summary.writeback`
- `diagnostic.writeback`
- `artifact.ref.writeback`

### 7.6 任务编排与生命周期管理层（Orchestration Service）

职责：

- 管理 task / agent 生命周期
- 维护任务状态机
- 维护父子任务关系
- 维护 coordinator / worker / verifier / teammate 关系
- 负责把治理命令分配给合适客户端执行

### 7.7 协作消息与邮箱层（Mailbox / Collaboration Service）

职责：

- 替代当前本地 `teammateMailbox.ts`
- 统一 direct message / control message / approval message
- 维护 read / unread / delivered / acknowledged 状态

### 7.8 Projection / Subscription 层

职责：

- 把权威状态转换成面向客户端的增量投影
- 支持 WebSocket push 与断线重连重放
- 支持局部 projection：team view / task view / mailbox view / governance view

### 7.9 事件记录与恢复层（Event Log / Snapshot / Recovery）

职责：

- append-only 事件记录
- 快照生成
- 快照之后事件回放
- 崩溃恢复与重建 projection

### 7.10 证据与附件层（Artifact / Evidence Service）

职责：

- 存储 `outputRef / artifactRef / evidenceRef`
- 记录 patch、日志、摘要、诊断附件的引用关系
- 仅在授权或需要时拉取原始内容

## 9. 信息同步边界裁决

### 9.1 按模式与工作区划分的总规则

#### 离线模式

- 不要求连接权威服务器
- 当前 `src/` 的 `AppState / tasks / teamContext / inbox` 继续作为本地真相
- 不存在服务端 `workspaceId` 绑定
- 所有任务、团队、消息和审批都只在本地闭环

#### 在线模式

- 必须先完成 `session.register`
- 必须绑定一个 `activeWorkspaceId`
- 若未绑定工作区，则在线模式仅能停留在“连接已建立、协作未激活”的状态
- 所有共享控制面对象必须归属某个 `workspaceId`

#### 工作区归属规则

- 个人工作区：默认承载用户自己的在线私有协作状态
- 共享工作区：承载多人共享 team/task/mailbox/governance
- 从个人工作区向共享工作区发布任务、摘要、证据或产物，必须通过核心治理请求完成，而不是客户端直接复制状态

### 9.2 必须同步到服务端的权威信息

| 信息类型 | 是否必须同步 | 说明 |
| --- | ---: | --- |
| 用户/客户端/session 身份 | 是 | 多端共享与路由基础 |
| 工作区本体与成员关系 | 是 | 在线模式的顶层命名空间与访问控制基础 |
| team roster / lead / member 关系 | 是 | 替代本地 `teamContext + TeamFile` |
| task 生命周期 | 是 | 任务状态必须有服务端唯一真相 |
| 审批与治理状态 | 是 | permission / plan / sandbox / shutdown |
| 控制消息与协作消息 | 是 | 替代本地 mailbox |
| 事件记录与快照版本 | 是 | 审计与恢复锚点 |

### 9.3 必须走受控回写的结果信息

| 信息类型 | 是否同步 | 同步方式 |
| --- | ---: | --- |
| task progress summary | 是 | 受控回写 |
| tool result summary | 是 | 受控回写 |
| transcript summary | 是 | 受控回写 |
| diagnostic summary | 是 | 受控回写 |
| outputRef / artifactRef / evidenceRef | 是 | 引用回写 |

### 9.4 默认不同步到服务端的信息

| 信息类型 | 是否同步 | 原因 |
| --- | ---: | --- |
| AsyncLocalStorage 内容本体 | 否 | 私有运行时上下文 |
| 本地完整文件树 | 否 | 非控制面真相 |
| 本地 secrets / token / `.env` | 否 | 安全边界 |
| tmux pane 内部屏幕缓冲本体 | 否 | 运行时细节 |
| REPL VM 对象本体 | 否 | 进程内实现细节 |

### 9.5 按需授权同步的信息

| 信息类型 | 默认策略 | 说明 |
| --- | ---: | --- |
| 文件片段 | 按需 | 用户授权或任务需要 |
| patch 内容 | 按需 | 建议先传 `patchRef` |
| 原始日志 | 按需 | 默认只传摘要/引用 |
| 完整 transcript | 按需 | 默认只上传 summary |

## 10. 同步协议设计

### 10.1 在线模式握手与工作区绑定

在线模式下建议采用两阶段握手：

1. `session.register`
   - 注册客户端、用户、协议版本、运行模式
   - 服务端返回：
     - `sessionId`
     - `personalWorkspaceId`
     - 可访问的共享工作区列表

2. `workspace.bind`
   - 将当前 session 绑定到某个 `activeWorkspaceId`
   - 绑定成功后，客户端开始接收该工作区的 team/task/mailbox/governance projection

离线模式下不要求这两步，客户端直接在本地离线工作区中运行。

## 12. 关键工作流

### 12.1 客户端启动与模式选择

1. 客户端启动。
2. 用户或配置决定当前进入：
   - `offline`
   - `online`
3. 若为离线模式，则直接启用本地控制面与本地运行时能力。
4. 若为在线模式，则进入 `session.register` 与工作区绑定流程。

### 12.2 在线模式连接与工作区绑定

1. 客户端启动。
2. 调用 `session.register`。
3. 服务端创建/恢复 `SessionAggregate`。
4. 服务端返回：
   - `sessionId`
   - `personalWorkspaceId`
   - 可访问的共享工作区列表
   - 初始 projection cursor
5. 客户端选择或恢复 `activeWorkspaceId`。
6. 调用 `workspace.bind`。
7. 服务端开始推送该工作区下的 team/task/mailbox/governance 投影。

### 12.3 创建团队

1. 客户端发 `team.create`。
2. 服务端分类为核心治理请求。
3. 进入裁决层。
4. 进入统一变更流水线。
5. 生成：
   - `TeamCreated`
   - `TeamLeadAssigned`
6. Projection 层把新 team 投影推回客户端。

### 12.4 派生 agent / teammate

1. 客户端发 `agent.spawn`。
2. 服务端生成 `TaskAggregate` 与成员关联。
3. 服务端将任务分派给目标客户端执行。
4. 目标客户端本地启动 runtime。
5. 本地 runtime 只回写 progress/result，不直接改权威状态。

### 12.5 任务进度与结果回流

1. 本地 agent 执行。
2. 客户端周期性发送 `task.progress.writeback`。
3. 服务端通过受控回写层更新 `TaskAggregate.progress`。
4. 完成后发送 `task.result.writeback`。
5. 服务端生成：
   - `TaskCompleted / TaskFailed / TaskKilled`
   - `TaskNotificationQueued`
6. Projection 层向相关客户端推送 `projection.task.notification`。

### 12.6 审批流

1. 某 agent 发出审批请求。
2. 服务端写入 `GovernanceAggregate(status=pending)`。
3. 服务端向 leader 推送 `projection.governance.request`。
4. leader 响应 `governance.respond`。
5. 服务端更新 `GovernanceAggregate`。
6. Projection 层通知请求方退出 `awaiting_approval`。

### 12.7 个人工作区到共享工作区的发布

1. 用户在个人工作区内完成任务、摘要、计划或产物准备。
2. 客户端发起 `workspace.publish_from_personal` 治理请求。
3. 服务端检查：
   - 目标共享工作区 membership
   - 发布对象类型与权限
   - 是否需要审批或改写
4. 通过统一变更流水线，将个人工作区中的可发布对象转换为共享工作区中的新对象或引用。
5. Projection 层向共享工作区成员广播新的变化。

### 12.8 离线转在线、恢复与重连

#### 离线转在线

1. 客户端处于离线模式并已有本地任务/摘要/产物。
2. 用户切换到在线模式。
3. 客户端完成 `session.register` 与 `workspace.bind`。
4. 用户可选择：
   - 不迁移任何本地状态
   - 只迁移摘要/证据引用
   - 把部分任务发布到个人工作区

#### 恢复与重连

1. 客户端断线。
2. 服务端保留 `SessionAggregate` 的最近状态与 projection cursor。
3. 客户端重连并附带 cursor。
4. 服务端增量补发事件；若 cursor 失效则强制 `projection.resync.required`。

## 13. 一致性、幂等与恢复策略

### 13.1 幂等

所有客户端命令必须带 `idempotencyKey`，服务端对以下操作做幂等：

- `team.create`
- `agent.spawn`
- `task.stop`
- `governance.respond`
- `message.send`
- `task.result.writeback`

### 13.2 投影一致性

客户端 `AppState` 不再是最终真相，只是：

- 最近一次投影
- 局部 UI 缓存
- 断线时的临时展示

若与服务端冲突，以服务端版本号为准。

### 13.3 恢复

第一阶段可先采用：

- append-only 事件表
- 定期快照 `TeamAggregate / TaskAggregate / GovernanceAggregate`
- 断线重连时按版本补发 projection

### 13.4 模式切换与工作区隔离

- 离线模式与在线模式之间不做隐式全量同步，必须通过显式迁移或发布动作进入在线控制面。
- 在线模式下，所有共享对象必须按 `workspaceId` 隔离，不能跨工作区直接复用状态对象。
- 个人工作区与共享工作区之间允许复制摘要、引用和经过批准的任务对象，但不允许客户端绕过服务端直接写入共享工作区真相。
- 工作区切换时，本地 `AppState` 应整体切换到新的 projection 流，而不是混合显示两个工作区的共享状态。

## 14. 安全与隐私边界

### 14.1 默认不上送的内容

- 本地 secrets / token / env
- 完整源码树
- 原始 REPL VM 内存对象
- tmux pane 完整缓冲
- 本地完整 transcript

### 14.2 上送时优先使用摘要/引用

优先同步：

- summary
- artifactRef
- evidenceRef
- patchRef

避免默认同步：

- 原始文件全文
- 原始 shell 全量输出
- 原始网页全文

### 14.3 审批必须服务端化

以下行为必须以服务端裁决结果为准：

- permission approval
- sandbox approval
- plan approval
- shutdown approval
- mode change

## 17. 最终结论

这套架构的核心不是“把所有 agent 都搬到服务器上跑”，而是：

> **把服务端建设为逻辑上全局唯一的权威控制平面，把当前 `src/` 改造成支持离线模式与在线模式双模运行的客户端；离线模式保持现有本地能力，在线模式则在权威服务器的个人工作区与共享工作区体系下运行，并由服务端统一收束共享状态、协作消息、任务生命周期、审批治理和审计恢复。**

需要特别明确的在线模式新增点有三类：

1. **在线模式必须新增的控制面能力**
   - `session.register / heartbeat`
   - workspace 目录、绑定、切换
   - server mailbox
   - server task registry
   - server governance
   - projection / resync
   - event log / snapshot / recovery

2. **在线模式必须服务端化的现有本地真相**
   - `AppState.tasks`
   - `AppState.teamContext`
   - `AppState.inbox`
   - `TeamFile`
   - `teammateMailbox.ts`
   - `task-notification`

3. **在线模式不应重写、而应继续复用的本地执行器**
   - Bash
   - REPL
   - 本地文件工具
   - worktree
   - tmux / iTerm teammate
   - in-process teammate
   - remote CCR 执行后端

一句话收束：

> **离线模式保留当前本地真相，在线模式新增服务端工作区、服务端任务真相、服务端邮箱、服务端治理和服务端恢复链路；执行仍在客户端，真相转移到服务端。**

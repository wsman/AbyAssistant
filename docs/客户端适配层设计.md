# 客户端适配层设计

> 本文档定义如何把当前 `src/` 改造成“支持离线模式 / 在线模式双模运行”的客户端，并在在线模式下接入权威服务器。重点不在服务端本身，而在客户端如何分层、如何兼容现有本地行为、如何逐步替换 `TeamFile / teammateMailbox / SendMessage / tasks / governance` 等本地控制面模块。

---

## 1. 文档目标

本文档回答以下问题：

1. 当前客户端应该增加哪些新的状态与抽象层。
2. 哪些现有模块在离线模式保持不变，哪些在在线模式下必须改走服务端。
3. 如何分阶段替换本地文件邮箱、任务注册表与审批治理。
4. 如何保证双模兼容，不破坏当前 `src/` 的本地能力。

---

## 2. 客户端适配层总体原则

### 2.1 双模兼容原则

客户端必须支持两种模式：

- `offline`：延续当前本地控制面
- `online`：连接权威服务器，使用服务端控制面

结论是：

> **不能把现有本地实现整体删掉，而应在其上方增加一层“适配层”，根据运行模式选择本地后端或服务端后端。**

### 2.2 运行时能力不动，控制面后端可切换

以下内容尽量保留原实现：

- 本地 Bash
- REPL VM
- tmux / iTerm2 pane
- in-process teammate
- 本地工具调用

需要切换的是“控制面后端”：

- team roster
- mailbox
- task registry
- governance
- projection state

### 2.3 AppState 在在线模式下降级为 projection cache

当前很多本地状态默认被当作真相。在线模式下必须重新定义：

- `AppState.tasks`：服务端 task projection cache
- `AppState.teamContext`：服务端 team projection cache
- `AppState.inbox`：服务端 mailbox projection cache

## 3. 客户端新增核心状态

建议在客户端状态层新增以下字段。

## 3.1 `runtimeMode`

```ts
type RuntimeMode = 'offline' | 'online'
```

作用：

- 决定当前控制面走本地还是服务端
- 所有适配器都以它为第一分流条件

## 3.2 `syncStatus`

```ts
type SyncStatus =
  | 'disconnected'
  | 'connecting'
  | 'registered'
  | 'workspace_bound'
  | 'synced'
  | 'stale'
  | 'error'
```

作用：

- 表示在线模式的同步健康状态
- UI 可据此显示是否处于服务端控制面下

## 3.3 `activeWorkspace`

```ts
type ActiveWorkspace = {
  workspaceId: string
  type: 'personal' | 'shared'
  name: string
}
```

作用：

- 指出当前在线模式下所绑定的工作区

## 3.4 `workspaceCatalog`

```ts
type WorkspaceCatalog = {
  personalWorkspaceId?: string
  sharedWorkspaces: Array<{
    workspaceId: string
    name: string
    role: 'owner' | 'admin' | 'member' | 'viewer'
  }>
}
```

作用：

- 用于工作区切换与共享协作入口展示

## 4. 适配层分层设计

建议把客户端新增三层抽象。

## 4.1 AuthorityClient

职责：

- 封装所有服务端 HTTP / WebSocket 通信
- 提供命令发送与事件订阅能力

建议接口：

```ts
interface AuthorityClient {
  registerSession(...): Promise<...>
  bindWorkspace(...): Promise<...>
  sendCommand(...): Promise<...>
  subscribe(...): AsyncIterable<ProjectionEventEnvelope>
  resync(...): Promise<...>
}
```

## 4.2 ControlPlaneAdapter

职责：

- 统一对外暴露“控制面动作”
- 内部根据 `runtimeMode` 分流到：
  - LocalControlPlaneAdapter
  - OnlineControlPlaneAdapter

例如：

- `createTeam()`
- `sendMessage()`
- `spawnTask()`
- `requestGovernance()`
- `ackMailboxMessage()`

## 4.3 ProjectionStore

职责：

- 把服务端 projection 写回本地 `AppState`
- 在在线模式下统一接管：
  - tasks
  - teamContext
  - inbox

## 5. 现有模块的改造策略

## 5.1 `teammateMailbox.ts`

### 现状

- 以本地 JSON 文件 + lockfile 方式承载 inbox

### 改造目标

- `offline`：继续使用当前实现
- `online`：改走服务端 mailbox API

### 建议做法

保留统一接口，例如：

```ts
interface MailboxBackend {
  readUnread(...): Promise<...>
  send(...): Promise<...>
  ack(...): Promise<...>
}
```

然后实现两个后端：

- `LocalFileMailboxBackend`
- `ServerMailboxBackend`

### 结论

`teammateMailbox.ts` 不应立刻删除，而应先被重构成**双后端门面**。

## 5.2 `SendMessageTool.ts`

### 现状

- 直接写本地 mailbox

### 改造目标

- `offline`：保持写本地 mailbox
- `online`：调用 `message.send` 命令

### 注意

- 在线模式下，不允许工具直接修改本地 inbox 真相
- 最终消息可见性必须来自服务端投影

## 5.3 `useInboxPoller.ts`

### 现状

- 轮询本地 inbox 文件并分类处理消息

### 改造目标

- `offline`：继续本地轮询
- `online`：改为订阅服务端 mailbox / governance / task notification 事件流

### 建议做法

把它重构成：

- `LocalInboxPoller`
- `OnlineInboxSubscriber`

然后由外层根据 `runtimeMode` 选择。

## 5.4 `TeamCreateTool.ts` / `TeamDeleteTool.ts`

### 现状

- 直接改本地 `teamContext` 和 team file

### 改造目标

- `offline`：继续本地创建/删除 team
- `online`：只发命令 `team.create` / `team.delete`

### 关键约束

- 在线模式下，本地不应直接创建最终 team 真相
- teamContext 只能等服务端投影回流后更新

## 5.5 `LocalAgentTask` / `RemoteAgentTask` / `InProcessTeammateTask`

### 现状

- 既承担本地 runtime 生命周期，又承担本地任务状态维护

### 改造目标

- `offline`：继续本地任务状态维护
- `online`：任务状态真相迁到服务端，这些模块只保留 runtime adapter 职责

### 在线模式下的新职责

- 接受服务端分派
- 启动本地执行
- 回写 `task.progress.writeback`
- 回写 `task.result.writeback`

## 5.6 `AppState.tasks`

### 现状

- 本地任务真相

### 改造目标

- `offline`：仍是本地真相
- `online`：变成 task projection cache

### 关键规则

- 在线模式下不允许本地直接把 task 置为 completed/failed/killed 最终状态
- 最终状态只能来自服务端 projection

## 5.7 `AppState.teamContext`

### 现状

- 本地共享团队视图

### 改造目标

- `offline`：继续本地维护
- `online`：变成 team projection cache

## 5.8 `AppState.inbox`

### 现状

- 本地待处理消息队列

### 改造目标

- `offline`：本地 inbox queue
- `online`：服务端 mailbox 投影缓存

## 6. 在线模式下的客户端主循环变化

在线模式下，客户端主循环应新增以下步骤：

1. `session.register`
2. 更新 `syncStatus=registered`
3. 拉取 `workspaceCatalog`
4. 选择或恢复 `activeWorkspace`
5. `workspace.bind`
6. 更新 `syncStatus=workspace_bound`
7. 建立 projection 订阅
8. 将 projection 写回本地 `AppState`
9. 本地运行时继续执行，但所有共享结果改为 writeback

---

## 7. 双模下的行为矩阵

| 能力 | offline | online |
|---|---|---|
| Team 创建 | 本地直接创建 | 发 `team.create` 命令 |
| SendMessage | 写本地 mailbox | 调 `message.send` |
| Inbox 消费 | 轮询本地文件 | 订阅服务端 mailbox |
| Task 真相 | 本地 `AppState.tasks` | 服务端 task projection |
| Team 真相 | 本地 `teamContext` | 服务端 team projection |
| 审批真相 | 本地桥接 | 服务端 governance |
| Runtime 执行 | 本地 | 本地或远端，但结果回写服务端 |

---

## 8. 推荐重构顺序

### Phase A：先引入双模状态与 AuthorityClient

目标：先不改业务，只补基础骨架。

包含：

- `runtimeMode`
- `syncStatus`
- `activeWorkspace`
- `workspaceCatalog`
- `AuthorityClient`
- `ProjectionStore`

### Phase B：替换 inbox 链路

目标：让多人协作先走服务端消息层。

优先改：

- `teammateMailbox.ts`
- `SendMessageTool.ts`
- `useInboxPoller.ts`

### Phase C：替换 task registry

目标：在线模式下让任务生命周期以上服务端为准。

优先改：

- `AppState.tasks`
- `LocalAgentTask`
- `RemoteAgentTask`
- `InProcessTeammateTask`

### Phase D：替换 teamContext 与治理流

优先改：

- `teamContext`
- `TeamCreateTool`
- `TeamDeleteTool`
- permission / sandbox / plan approval / shutdown

---

## 9. 兼容性要求

### 9.1 不破坏离线行为

- 若服务端不可用，客户端仍必须能以 `offline` 运行
- 不允许因为在线改造而导致当前本地 CLI 无法单独使用

### 9.2 在线失败要可回退

- `online` 注册失败时，可回退为 `offline`
- 工作区绑定失败时，不应让本地 runtime 崩溃

### 9.3 渐进替换，不一次性推倒

- 同一模块先做双后端门面，再逐步把默认在线路径切到服务端

## 11. 最终结论

客户端适配层的本质不是“把当前代码改成只能在线跑”，而是：

> **在保留现有 `src/` 离线本地能力的前提下，增加一层可切换的控制面适配层，使在线模式下的 team、task、mailbox、governance 与 projection 统一对接权威服务器，而离线模式下仍可继续使用当前本地实现。**

因此，客户端改造的正确策略是：

1. 先补双模状态与服务端通信抽象；
2. 再替换 inbox；
3. 再替换 task registry；
4. 最后统一 teamContext 与 governance；

而不是一开始就把所有本地控制逻辑整体删除。
